## 1. 프로젝트 개요

### 배경 및 문제 정의

금리 인상, 대출 규제, 조정대상지역 지정 등 부동산 시장에 영향을 주는 정책 이벤트는 반복적으로 발생한다. 그러나 이러한 이벤트 이후 **어느 지역이, 언제, 얼마나 반응했는지**를 데이터로 직접 확인할 수 있는 수단은 사실상 없다.

기존 서비스의 한계는 명확하다. 뉴스 기사는 단편적 사실만 전달하고, 시세 조회 서비스는 현재 가격만 보여주며, 공공 데이터는 존재하지만 일반 사용자가 직접 분석하기에는 접근 장벽이 높다. 특히 부동산 시장은 지역별 반응 속도와 강도가 다르기 때문에 단순 평균 가격만으로는 시장 흐름을 체감하기 어렵다.

### 프로젝트 정의

> 과거 아파트 매매 실거래 데이터를 기반으로, 경제·정책 이벤트 이후 지역별 가격·거래량 변화가 어떤 순서와 강도로 확산되었는지 분석하여 시각적인 자료를 제공하며, 분석된 패턴을 기반으로 유사 이벤트에 대한 시장 참여자 반응을 시나리오 형태로 탐색하는 시스템
> 

### 핵심 분석 개념

| 개념 | 정의 |
| --- | --- |
| **반응 시차 (Lag)** | 이벤트 발생 후 해당 지역이 임계 변화율을 처음 초과한 시점까지의 차이 |
| **충격 전파 (Diffusion)** | 선행 반응 지역에서 후행 반응 지역으로 가격·거래량 변화가 확산되는 경로 |
| **풍선효과 (Displacement)** | 규제 지역 거래 감소 → 인접·외곽 지역 거래 증가로 수요가 이동하는 패턴 |

### 프로젝트 범위 및 제한 사항

본 시스템은 미래 가격을 예측하는 시스템이 아니다. 과거 데이터에서 관측된 반응 패턴과 전파 구조를 기반으로 가능한 시장 흐름을 탐색하는 **시나리오 탐색기**로 정의한다. 지역 간 전파 구조는 시차 및 변화율 상관 구조 기반 추정이며, 인과관계가 아닌 상관관계임을 결과 화면에 명시한다.

## 2. 분석 대상 범위

### 지역

| 구분 | 대상 |
| --- | --- |
| 서울특별시 | 전 구 (25개 자치구) |
| 수도권 거점 | 수원, 성남, 화성, 용인, 광명, 고양, 하남, 과천, 김포 |

서울 규제 강화 시 수도권 풍선효과가 가장 빈번하게 관측되는 지역군으로 구성했다. 강남 → 송파 → 성남 → 용인 방향의 전파 경로가 과거 데이터에서 반복적으로 나타난 지역들을 우선 포함한다.

### 분석 기간

API 트래픽 한도 내에서 최대한 넓은 구간을 확보한다. 목표는 2000년 이후 전 구간이며 최소 2025년까지 커버한다. 저금리 급등기 → 금리 인상 및 거래절벽 → 회복기 사이클을 모두 포함하는 구간을 확보하는 것이 핵심이다. 실제 수집 가능 범위는 API 콜 수 계산 후 1단계에서 확정한다.

### 자산 유형

아파트 매매 실거래가를 1차 분석 대상으로 한다. 전월세 데이터는 3순위 확장 기능으로 분류한다.

정책 발표 이후 내 지역이 어떤 영향을 받는지, 다른 지역은 어떻게 움직였는지를 뉴스가 아닌 데이터로 확인하고 싶은 사용자다. 전문 지식 없이도 지도와 타임라인으로 흐름을 파악하는 것이 핵심 니즈다.

주요 관심사: 정책 발표 이후 어떤 지역이 먼저 반응했는가 / 우리 지역이 얼마나 영향을 받았는가 / 풍선효과가 어느 방향으로 흘렀는가

**분석 학습자 — 데이터 분석·정책 분석 관심층**

주요 관심사: 정책 이벤트 기반 데이터 분석 과정 / 시계열 및 공간 데이터 시각화 구현 방식 / 공공 데이터 ETL 구조

### 이벤트 데이터 관리 방식

경제지표(금리, CPI 등)는 ECOS API와 R-ONE API로 자동 수집한다. 정책 이벤트(규제 지정, 공급 발표 등)는 관리자가 직접 이벤트명·발생일·유형을 등록하는 방식으로 관리한다. 두 방식을 병행하여 정량 지표와 정성 이벤트를 함께 분석 기준으로 활용한다.

    ## 주요 함수 예시
    
    ### 1. Baseline 계산 (G4-01)
    
    ```python
    def compute_baseline(df, event_date, windows=6):
        start = pd.to_datetime(event_date) - pd.DateOffset(months=windows)
        end = pd.to_datetime(event_date) - pd.DateOffset(days=1)
        snippet = df[(df.contract_month >= start) & (df.contract_month < end)]
        return snippet.groupby('gu_code')['avg_price'].mean()
    ```

    def compute_change_rate(pre, post):

    ### 3. 반응 시점 탐지 (G4-05) — ruptures
    
    ```python
    import ruptures as rpt
    
    def detect_change_point(series, event_date, threshold_pct=2.0):
        mask = series.abs() > threshold_pct
        if mask.any():
            return mask[mask].idxmax()
    
        # ruptures 사용
        algo = rpt.Pelt(model="rbf").fit(series.values.reshape(-1,1))
        bkps = algo.predict(pen=10)
        for bkp in bkps:
            date = series.index[bkp-1]
            if date > pd.to_datetime(event_date):
                return date
        return None
    ```

    def compute_impact_score(price_change, volume_change, w1=0.6, w2=0.4):
        return w1 * abs(price_change) + w2 * abs(volume_change)
    ```
    
    ### 5. 선행·후행 순위 (G4-10)

    def rank_regions(lag_series, impact_series):
        df = pd.DataFrame({'lag': lag_series, 'impact': impact_series})
        df['lag_rank'] = df['lag'].rank(ascending=True)
        df['impact_rank'] = df['impact'].rank(ascending=False)
        df['combined_rank'] = df['lag_rank'] + df['impact_rank']
        return df.sort_values('combined_rank')
    ```

### G7. 시나리오 탐색기

### 목적 및 포지셔닝

G7은 분석 결과(G1~G6)를 숫자와 차트로만 보여주는 것에서 나아가, **시장 참여자 관점으로 번역하여 직관적으로 탐색**할 수 있도록 하는 해석 레이어다.

"강남구 거래량 -18%"라는 수치가 실제로 시장 참여자들에게 어떤 행동 변화를 유발했는지, 그리고 그 행동이 다른 참여자와 인접 지역에 어떤 연쇄 반응을 만들어냈는지를 페르소나 기반으로 탐색한다.

본 기능은 시뮬레이션이 아니다. LLM이 창발적 결과를 생성하거나 미래를 예측하지 않는다. **과거 분석 수치를 기반으로 규칙이 state를 변화시키고, LLM은 그 변화를 사람 언어로 서술하는 역할**만 담당한다. 결과는 재현 가능하며 분석 수치에서 벗어나지 않는다.

### 전체 구조

```
[G4~G6 분석 결과]
        ↓
  수치 기반 파라미터 변환
        ↓
[에이전트 초기 state 설정]
  - 페르소나별 초기 confidence / mobility / 관망 확률
  - 지역별 반응 시점 지연 설정
        ↓
라운드 시작
  → 규칙 기반 에이전트 행동 결정
  → 시장 state 변화 계산
  → LLM: state 변화를 페르소나 언어로 서술
        ↓
다음 라운드: 변화된 state 기반 행동 재계산
        ↓ (N라운드 반복)
[시나리오 결과 출력]
  - 페르소나별 반응 흐름
  - 시장 분위기 변화 추이
  - 지역별 관심도 이동
  - 요약 리포트
```

### 분석 결과 → 초기 state 변환 예시

| 분석 결과 | 에이전트 초기 state 반영 |
| --- | --- |
| 강남구 거래량 -18%, 변화율 -3.2% | 투자자 confidence 하향, 매물 출현율 상향 |
| 강남 → 송파 lag 1개월 | 송파 에이전트 반응 시점 1라운드 지연 |
| 외곽 지역 거래 증가 선행 | 외곽 실수요자 mobility 가중치 증가 |

### 시장 state 변수 및 연쇄 구조

| state 변수 | 변화 조건 | 영향 받는 에이전트 행동 |
| --- | --- | --- |
| 투자자 confidence | 거래량 감소 / 가격 하락 | 매물 출현율 증가 |
| 시장 거래량 | 매물 증가 / 수요 감소 | 중개사 호가 조정 확률 증가 |
| 시장 불안 심리 | 거래량 급감 / 금리 상승 | 실수요자 관망 확률 증가 |
| 외곽 관심도 | 핵심 지역 규제 강화 | 실수요자 지역 이동 확률 증가 |

### 에이전트 페르소나 구성

Nemotron-Personas-Korea 데이터셋의 직업(job), 자산 수준(asset_level), 주거 상태(housing_status) 필드를 조합하여 페르소나를 구성한다. 구성된 페르소나는 에이전트 초기 시스템 프롬프트로 주입된다.

| 페르소나 | 데이터셋 필드 조합 | 핵심 행동 변수 |
| --- | --- | --- |
| 투자자 | job=사업가/자영업, asset_level=상, housing_status=다주택 | confidence, 매물 출현율 |
| 실수요자 | job=직장인, asset_level=중, housing_status=무주택 | 관망 확률, 지역 이동 확률 |
| 임차인 | housing_status=임차, asset_level=하 | 월세 전환 민감도 |
| 공인중개사 | job=중개업 | 호가 조정 확률, 문의량 |
| 신혼부부 | age=30대 초반, housing_status=무주택 | 외곽 이동 민감도 |

### MiroFish 참조

G7의 UX 구조는 MiroFish(멀티에이전트 시뮬레이션 오픈소스)의 에이전트 대화 흐름 시각화 방식을 참조하여 설계한다. 단, 시뮬레이션 엔진 자체는 직접 구현하며 MiroFish 코드를 그대로 사용하지 않는다. MiroFish 구조를 분석하여 핵심 UX 개념만 차용하고 분석 결과 기반의 state 관리 방식은 본 프로젝트에 맞게 독자 설계한다.

### G7 요구사항

- 규칙 정의(수정 예정)
    
    지금 기획서의 state 변수 표는 방향만 있고 수치가 없다. 이걸 구현 가능한 수준으로 채우면 이렇게 된다.
    
    ### State 전이 규칙 예시
    
    `IF 지역 거래량 변화율 < -10%
      THEN 투자자.confidence -= 0.2
    
    IF 투자자.confidence < 0.4
      THEN 투자자.매물출현율 += 0.15
    
    IF 시장.매물출현율 > 0.6 AND 실수요자.관망확률 > 0.5
      THEN 중개사.호가조정확률 += 0.1
    
    IF 핵심지역.규제강화 = TRUE
      THEN 실수요자.외곽이동확률 += 0.25`
    
    숫자들이 지금은 임의처럼 보이는데, 정당화하는 방법이 있다.
    
    **실제 이벤트 역산 방식:** 2017년 8.2대책 이후 강남구 거래량이 실제로 약 40% 급감했고, 같은 시기 경기 외곽(화성, 김포) 거래량이 증가했다. 이 실제 수치를 state 전이 결과로 "재현"되도록 파라미터를 역산하면, 그 파라미터가 임의가 아니라 **관측값 기반 캘리브레이션**이 된다.
    
    이게 G7의 핵심 설득 논리가 될 수 있다. "우리 규칙은 8.2대책 이후 실제 시장 반응을 기준으로 캘리브레이션됐다"고 하면, 시뮬레이션 임의성 비판을 크게 줄일 수 있다.
    
    ### LLM 재현성 문제 해결
    
    위에서 지적한 "재현 가능"이라는 표현 문제인데, 현실적인 해결책은 두 가지다.
    
    - **temperature=0 고정** + 프롬프트에 state 수치를 정확히 주입 → 출력 변동 최소화
    - 기획서 표현을 "재현 가능"에서 **"분석 수치 범위 내에서 서술"** 로 바꾸고, LLM이 수치를 임의로 이탈하지 못하도록 프롬프트에 "다음 수치를 반드시 포함하여 서술하라"는 앵커를 넣는다

| **3단계** | 이벤트 관리 기능, 분석 엔진 (변화율·시차·충격 강도) | 2주 | 핵심 분석 파트. 일정 초과 시 G7 범위 축소 |
| **4단계** | 지도 시각화, 타임라인 슬라이더, 전파 애니메이션 | 1~2주 |  |
| **5단계** | 자동 리포트 생성, LLM 연동, 시드 문서 자동화 | 1주 |  |
| **6단계** | 시나리오 탐색 엔진, 에이전트 state 설계, 페르소나 시각화 | 1~2주 | G1~G6 완료 후 착수 |
